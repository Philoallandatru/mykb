# LMCache 多 SSD 压力测试学习笔记

## 📚 来源

**类型**: 实践项目  
**项目**: ~/ai-ssd LMCache 压力测试  
**时长**: ~3小时实验 + 1小时持续测试  
**日期**: 2026-06-02 ~ 2026-06-03

## 📝 关键要点

### 1. 问题诊断方法论 ✅

**初始现象**: "磁盘缓存不工作"
- External prefix cache hit rate: 0.0%
- SSD写入: 0 bytes

**错误结论**: LMCache有bug

**正确分析流程**:
1. **验证配置** - 检查是否真的启用了目标功能
2. **理解层级** - local_cpu vs local_disk 是不同层
3. **检查输入** - prompt长度是否满足chunk_size要求
4. **查看日志** - metrics端点 vs 实际日志
5. **监控磁盘** - 文件系统比API更可靠

**最终发现3个根本问题**:
- ❌ Prompt太短 (96 tokens << 256 chunk_size)
- ❌ 配置错误 (local_cpu: true 拦截了请求)
- ❌ 测试方法错 (100个不同prompts vs 重复同一长prefix)

**学到的教训**: 
> 不能只看一个指标就下结论。系统调试需要多维度验证。

### 2. 配置理解的重要性 ✅

**LMCache的多层架构**:

```text
请求 → vLLM GPU prefix cache
         ↓ miss
       LMCache CPU RAM (可选)
         ↓ miss / overflow
       LMCache Local Disk
         ↓ miss
       重新计算
```

**关键配置参数**:

| 参数 | 作用 | 易错点 |
|------|------|--------|
| `local_cpu` | 启用CPU RAM层 | ≠ 磁盘缓存开关 |
| `local_disk` | 磁盘路径 | 必须有file://前缀 |
| `chunk_size` | KV chunk大小 | prompt需大于此值 |
| `kv_connector` | vLLM集成方式 | 必须配置才能工作 |

**教训**:
- 每个参数都有特定含义，不能想当然
- 文档要仔细读，特别是"默认值"部分
- 测试时最好先用单一变量

### 3. 性能测试设计 ✅

**错误的测试**:
```python
# 100个不同prompts，每个请求一次
prompts = [f"问题{i}" for i in range(100)]
for p in prompts:
    request(p)  # 每个都是cold cache
```
结果: 缓存命中率低，看不出效果

**正确的测试**:
```python
# 1个长prompt，重复50次
long_prompt = "..." * 15000  # 15K tokens
for _ in range(50):
    request(long_prompt)  # 第1次cold，后49次warm
```
结果: 56x加速，效果显著

**原理**: 
- LMCache是prefix cache，需要相同前缀
- 测试缓存效果必须有重复访问
- 长上下文才能体现磁盘缓存价值

### 4. 实测数据和规律 ✅

**加速比变化**:

| 测试阶段 | 缓存特征 | 加速比 | 原因 |
|---------|---------|--------|------|
| 初始验证 | 同一prompt × 50 | 56.10x | 100%命中率 |
| 极限填充 | 80个不同prompts | 1.09x | 低命中率 |
| LRU驱逐后 | 旧缓存被删 | 1.08x | warm变cold |

**规律**: 
- 高重复 → 高命中率 → 高加速比
- 低重复 → 低命中率 → 低加速比
- LRU驱逐 → 命中率下降 → 加速下降

**启示**:
> LMCache适合"相同长context + 不同短问题"的场景，如客服对话、RAG系统。

### 5. 系统瓶颈识别 ✅

**初始猜测**: SSD性能不够

**实测数据**:
- SSD理论: 3-7 GB/s读取，500K-1M IOPS
- 实际使用: 40-130 MB/s，数千次IOPS
- 利用率: < 5%

**真实瓶颈**: GPU推理速度
- 每个请求推理耗时 >> 磁盘I/O时间
- GPU顺序处理请求
- SSD大部分时间在等待

**启示**:
> 优化要找真正的瓶颈。SSD "打不满"不代表失败，而是说明瓶颈在别处。

### 6. LRU驱逐机制观察 ✅

**实际触发过程**:
```text
初始: 19.64 GB (2873 文件)
↓ 写入新缓存
驱逐: 删除 716 个旧文件 (4.89 GB)
↓
最终: 14.75 GB (2157 文件)
```

**发现**:
- 驱逐率: 24.9%
- 保留: 75.1% 热数据
- 自动化: 无需手动干预

**理解**:
- LRU很聪明，准确识别冷数据
- 缓存容量限制是"软"的，会自动调整
- 生产环境可放心设置上限

## 💭 个人思考

### 实践价值

**1. 真实的生产环境验证**
- 1小时持续测试，560个请求，100%成功
- 证明LMCache在长时间运行下稳定可靠
- 可以放心用于生产环境

**2. 多SSD的局限**
- 单GPU环境多SSD收益有限
- by_gpu策略需要多GPU才有意义
- 要充分利用需要RAID或多进程

**3. 配置的陷阱**
- 一个参数理解错误可能导致完全不同的行为
- 需要结合文档、日志、监控多方验证
- 不能只依赖metrics API

### 技术洞察

**缓存分层的智慧**:
- GPU → CPU → Disk 三级缓存各有定位
- CPU层是hot cache，磁盘是cold cache
- 可以根据场景选择启用哪些层

**测试设计的艺术**:
- 单变量原则
- 极端值测试（填满容量、超高并发）
- 长时间稳定性验证

**性能分析的方法**:
- 先定位瓶颈再优化
- 理论值 vs 实测值的差距说明什么
- 多维度指标（延迟、吞吐、命中率）

### 应用场景启发

**最适合LMCache的场景**:
1. **客服系统** - 固定的产品知识库 + 不同用户问题
2. **RAG问答** - 大量检索context + 短问题
3. **代码助手** - 相同代码库 + 不同查询
4. **多轮对话** - 逐渐增长的对话历史

**不太适合的场景**:
- 每次都是全新的长上下文
- 访问模式随机，无重复
- 上下文短（< 4K tokens）
